## 修复：DGS/DCPN3M 等 FRED 序列 previous 为 null（bug）
- 用户发现 DGS1/DGS3/DGS10 `previous=null` 但 `value` 有数（as_of=2026-07-06），问是否符合预期。结论：**不符合，是真实 bug**。
- 根因：`scripts/fetchers.py` 的 `fetch_fred_series` 走 FRED API 路径时 `limit=2, sort_order=desc`。紧邻市场假期（7/3 独立日）时 FRED 会给假期日期发一条 `value="."` 的 observation 占位，把真实上一交易日（7/2）顶出 limit=2 窗口；"." 被 `safe_float` 过滤后只剩 1 条有效行 → `previous=None`。系统性影响所有 `fetch_fred_series` 序列（DGS*/DCPN3M/DFII10/DTB3），NY Fed 来源序列不受影响。
- 修复：`limit=2` → `limit=30`（过滤 "." 后取最后两条有效行即可；任何假期空档都远小于此）。CSV 回退路径（无 key）本就拉整个窗口，不受此影响。
- 验证：重跑 `run.py` 后 DGS1/3/10 previous 恢复为 7/2（3.96/4.16/4.49），全部 FRED 序列无 value 有数但 previous=null。DCPN3M 仍滞后到 2026-06-26（自身发布慢），属数据滞后非本次 bug。
- 已 commit+push（942ba56）。后续每次假期后首跑此 bug 不再出现。
